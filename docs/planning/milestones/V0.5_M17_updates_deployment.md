# Milestone M17 — Updates & deployment

**Closes:** v0.5.2  **Branch:** `0.5-dev`  **Theme:** the §15 deployment lifecycle
-- keep a fielded station current and recoverable. (Design proposal "M15".
F5 support bundle, F11 updates/SD image.)

## Deliverables

| # | DESIGN ref | Deliverable |
| -- | -- | -- |
| D1 | §15 updates (channels, rollback) | update runner: pinned **channels** (stable/beta), apply + **rollback** to the prior version; surfaced in Settings -> Software (scaffolded in M15) |
| D2 | §15 config backup/restore | full **backup/restore** bundle (config + optional data manifest); scheduled backups to the configured location |
| D3 | §15 SD image | reproducible **SD-card image** build for Raspberry Pi OS (extends the M7 `.deb`) |
| D4 | F5 support bundle | **support-bundle export**: logs + versions + config (secrets redacted) as one downloadable archive (replaces the legacy TeamViewer step) |

## Notes

- The update runner touches the host; least-privilege hook + admin-gated, like
  the M15 reboot affordance.
- Support bundle must **redact secrets** (the encrypted credentials, secret_key)
  -- assert this in tests.

## Acceptance

- Update apply + rollback works against a staged channel (fake-channel test).
- Backup -> wipe -> restore reproduces the station.
- The SD-image build script produces a flashable image (CI builds the recipe).
- Support bundle exports and is secret-free.
- Quality gate green; SNR clean; tag v0.5.2.

## Out of scope

Resilience model (M16); wizard (M18).
