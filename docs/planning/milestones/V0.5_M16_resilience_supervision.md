# Milestone M16 — Resilience & supervision

**Closes:** v0.5.1  **Branch:** `0.5-dev`  **Theme:** the §14a/§12a robustness
model -- *degrade, don't die; never lose un-uploaded science data; isolate
faults; always alert*. (Design proposal "M14"; the M11 watchdog was the first
slice, this is the deep model. F10.)

## Deliverables

| # | DESIGN ref | Deliverable |
| -- | -- | -- |
| D1 | §5/§14a acquisition isolated as its own supervised process | run acquisition in a **separate supervised process** (not in-app threads), supervised by systemd; the web app stays up if acquisition restarts |
| D2 | §14a failure-mode matrix | encode the **degrade-don't-die policy matrix** (disk full, receiver gone, clock bad, upload backlog, web down) with defined responses; each fault isolated |
| D3 | §14a "always alert" | **alert channels**: email (SMTP) + webhook; alert routing for the health alerts already computed (M4) |
| D4 | §12a timing | finer **drift-gating** (thresholds + per-class timestamping model) beyond the binary M11 clock gate |

## Notes

- Process isolation changes how the recorder is launched -> ADR (acquisition
  IPC/contract); the testable `tick`/`record` units stay pure.
- Alert channels are an existing design extension point ("Alert Channel" plugin,
  §5a) -> implement behind that contract + a fake.

## Acceptance

- Killing the acquisition process does not take down the web app; it is restarted
  and the gap is alerted (fake-supervisor test).
- The failure matrix is data-driven and unit-tested per fault.
- An alert fires to a fake email + fake webhook channel.
- Quality gate green; SNR clean; tag v0.5.1.

## Out of scope

Update lifecycle (M17); GPS/PPS hardware timing (F3, backlog).
